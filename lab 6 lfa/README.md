## Introduction: What Are Parsing and Abstract Syntax Trees?

Parsing is the process of analyzing a sequence of tokens (generated by a lexer) to determine its syntactic structure according to the rules of a formal grammar. Unlike lexical analysis, which breaks source text into tokens, parsing connects these tokens into a hierarchical tree that represents how they relate to each other. The result of parsing is typically a parse tree, which explicitly shows the grammatical structure including all intermediate rules of the grammar.

An Abstract Syntax Tree (AST) is a specialized form of parse tree that retains only the essential structural information while omitting unnecessary syntactic details. Whereas a parse tree might include every rule application (even trivial ones), an AST abstracts away such redundancy and represents only the constructs that matter semantically. For example, an AST for an expression `(3 + 4) * 5` represents the fact that multiplication is applied to an addition and a literal, but it doesn't explicitly show every grouping rule or precedence rule that the parser used. ASTs are preferred in compiler construction and program analysis because they are more compact and directly represent the meaning of the source code.

## Relationship Between Lexer and Parser

This laboratory extends the lexer from Lab 3, which tokenizes source code into a flat sequence of tokens. The lexer identified token types using regular expressions—for example, matching patterns like `\d+` to recognize integers or `[A-Za-z_][A-Za-z0-9_]*` to recognize identifiers. However, tokenization alone does not capture how tokens combine into meaningful units. The parser takes the token stream and applies grammar rules to detect phrases and statements. In practice, lexer and parser are separate but sequential stages: the lexer runs first and produces tokens, then the parser consumes those tokens to build the AST.

## Key Improvements in This Laboratory

This lab extends Lab 3's lexer in several critical ways. First, a proper `TokenType` enum was introduced to replace string-based token types. Rather than using raw strings like `"IDENTIFIER"` or `"INT"`, the lexer now produces tokens with enum values (`TokenType.IDENTIFIER`, `TokenType.INT`), providing type safety and clarity. The lexer still uses regular expressions to identify token types—maintaining the dynamic approach from Lab 3—but now categorizes them into a structured enum.

Second, comprehensive AST node classes were designed to represent the syntactic structures of the language. These include expression nodes (`IntLiteral`, `BinaryOp`, `FunctionCall`, etc.), statement nodes (`VariableDeclaration`, `IfStatement`, `ReturnStatement`, etc.), and a root `Program` node. Each node type directly corresponds to a language construct, making the tree structure intuitive and easy to traverse.

Third, a recursive-descent parser was implemented to extract syntactic information from the token stream. The parser respects operator precedence (for example, multiplication binds tighter than addition), handles grouping with parentheses, and correctly parses complex constructs like function definitions, conditional statements, and array/map literals. The parser uses well-established techniques such as precedence climbing for binary operators and lookahead for statement disambiguation.

## Technical Approach: Recursive-Descent Parsing

The parser is implemented using recursive descent, a top-down parsing technique where each grammar rule is represented as a function. The parser maintains a current position in the token stream and implements functions for each grammatical level: `_parse_expression()` for full expressions, `_parse_additive()` for plus/minus operations, `_parse_multiplicative()` for multiply/divide operations, and so forth. This structure naturally enforces operator precedence—lower-precedence operators are handled at higher levels in the recursion, ensuring they bind less tightly.

When the parser encounters a choice (such as whether an identifier is part of an assignment or just a variable reference), it uses lookahead to peek at upcoming tokens without consuming them. For instance, if the parser sees an identifier followed by an assignment operator, it knows to parse an assignment statement; otherwise, it treats the identifier as part of an expression. This approach is simple to understand and implement, though it is limited to LL(1) or LL(k) grammars (languages where the parser can determine what to do with a limited lookahead).

## Challenges Addressed

One challenge was designing an AST representation that captures all necessary information without becoming overly complex. The solution was to keep node types focused and specific—separate classes for different constructs rather than generic `Node` classes with type fields. Another challenge was correctly implementing operator precedence in the recursive-descent parser; this was solved by using the standard precedence hierarchy (alternation, assignment, logical-or, equality, comparison, addition, multiplication, unary, postfix, primary) reflected in the function call hierarchy.

A third challenge was ensuring that the parser could handle all the language constructs present in the lexer's token set, including advanced features like map literals and array indexing. The solution involved careful consideration of which constructs should be expressions versus statements, and how to represent complex constructs like function calls and array access as postfix operators applied to primary expressions.
